# Action permissions, knowledge audit, and zero-width detection for Agent Guard (v2)

The agent-guard system gains read-only mode with granular action permissions, a knowledge-content auditor blocking prompt injection in knowledge entries, and zero-width Unicode detection in the input sanitizer. The v1 review raised 5 issues; 4 were addressed (ActionType validation added, `act as` regex narrowed to directive forms, zero-width regex extracted to shared constant, enforcement demonstrated in example-usage.ts). The admin-bypass omission was an intentional design choice. One new bug emerges from the shared regex approach.

**Watch for:** The shared `ZERO_WIDTH_CHARS_PATTERN` is a stateful `/g` regex used as a module-level constant — successive `.test()` calls across invocations will produce intermittent false negatives as `lastIndex` drifts (confirmed). The `system\s*:` pattern in the knowledge auditor remains broad enough to reject entries that document configuration formats or reference system labels (likely).

**Verdict**: NEEDS_CHANGES

## High-level view

The zero-width regex deduplication (issue #4 from v1) introduced a correctness bug: a `/g`-flagged regex stored as a module-level constant retains `lastIndex` between calls. Both the sanitizer and auditor use `.test()` on this shared regex, meaning the second call in the same process tick (or any subsequent call without a reset) may silently skip detection. This is the only blocking concern.

The `act as` pattern was tightened to a directive-anchored form, eliminating product-description false positives. The `system\s*:` pattern was not similarly narrowed and could still reject entries containing benign technical references like "Operating system: Windows" — minor given the Indonesian e-commerce context, but worth flagging since the auditor is a hard gate with no override.

## Stateful global regex causes intermittent detection failure

`ZERO_WIDTH_CHARS_PATTERN` is defined as:

```typescript
export const ZERO_WIDTH_CHARS_PATTERN =
  /[\u200B\u200C\u200D\uFEFF\u202A-\u202E\u2060\u2066-\u2069]/g;
```

In `input-sanitizer.ts`, the usage is:

```typescript
if (ZERO_WIDTH_CHARS_PATTERN.test(sanitizedInput)) {
  matchedPatterns.push("zero-width characters");
  sanitizedInput = sanitizedInput.replace(ZERO_WIDTH_CHARS_PATTERN, "");
}
```

After `.test()` finds a match, `lastIndex` advances past it. The subsequent `.replace()` starts from that advanced position, potentially missing characters before the index. On the next invocation of `sanitizeInput`, `lastIndex` is wherever the previous call left it — making behavior depend on the content of previous calls.

In `knowledge-auditor.ts`, the same constant is called with `.test()` only. After a successful match, `lastIndex` advances. If `auditKnowledgeContent` is called again immediately (e.g., batch-validating multiple entries), the second call may fail to detect zero-width characters that appear before the stale `lastIndex`.

The fix is either: (a) reset `lastIndex = 0` before each `.test()` call, or (b) store the pattern without `/g` and construct a `/g` variant only where `.replace()` needs it. Option (b) is cleaner — the constant stays a detection pattern, call sites choose whether to scan globally.

## `system\s*:` residual false-positive surface

The `system:` pattern remains `/system\s*:/i`, matching any occurrence of "system" followed by a colon. In a knowledge base documenting configuration formats or referencing English technical terms, this triggers on benign content. The risk is lower in Indonesian e-commerce (where "sistem" is more common), but entries quoting technical references would be rejected with no override path. An anchor like `/^system\s*:/im` (line-start only) would reduce false positives while still catching `system: you are a helpful assistant` injection patterns.
