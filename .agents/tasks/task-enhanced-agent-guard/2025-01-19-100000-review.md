# Action permissions, knowledge audit, and zero-width detection for Agent Guard

The agent-guard system gains three new capabilities: a read-only mode with granular action-level permissions, a knowledge-content auditor that blocks prompt injection in knowledge-base entries, and zero-width Unicode character detection in the input sanitizer. The Prisma schema, settings API, and UI all grow to accommodate the new `readOnlyMode` and `allowedActions` fields. The knowledge audit is wired into the POST endpoint as a hard gate before database writes.

**Watch for:** The `act as` regex in the knowledge auditor is overly broad and will reject legitimate product descriptions (confirmed). The `allowedActions` validation in the API doesn't check whether values are valid `ActionType` members, allowing arbitrary strings to be persisted (confirmed). The `isActionAllowed` utility is defined but never called at any enforcement point in this diff (confirmed).

## High-level view

The action-permissions module introduces a type-safe `isActionAllowed` function and an `ActionPermissionConfig` interface, but the enforcement point is missing from this change. No API route or agent pipeline calls `isActionAllowed` before performing a write action, so the permission toggles in the UI are cosmetic until a consumer is added.

The knowledge auditor scans content for prompt injection patterns before allowing writes to the knowledge base. Its pattern set includes overly broad matches ("act as", "system:") that will generate false positives on legitimate Indonesian customer-service content. The auditor is integrated as a hard 400 rejection, offering no admin override.

Zero-width character detection is added to both the input sanitizer (strips and flags) and the knowledge auditor (rejects). The regex is identical in both modules, creating a maintenance concern if one is updated without the other.

The PATCH API does not validate that `allowedActions` strings are members of the known `ACTION_TYPES` set, allowing arbitrary action names to be saved to the database.

<summary>Issues (5)</summary>

1. **`isActionAllowed` never enforced** — The permission utility exists but is not called anywhere in the request lifecycle. Wire it into the agent action pipeline or the relevant API routes so the permission toggles actually block actions.
2. **`act as` false positives** — The regex `/act\s+as/i` matches normal Indonesian and English phrases (e.g., product descriptions like "can act as a power bank"). Use a more targeted pattern or require surrounding injection context.
3. **No `ActionType` validation on save** — The PATCH endpoint validates that `allowedActions` is a JSON array of strings but doesn't check membership against `ACTION_TYPES`. Invalid action names persist silently.
4. **Duplicated zero-width regex** — The same character class exists in both `input-sanitizer.ts` and `knowledge-auditor.ts`. Extract to a shared constant so future updates don't diverge.
5. **Knowledge audit has no bypass for admins** — The auditor is a hard gate. If an admin legitimately needs to store a knowledge entry containing "system:" (e.g., documenting the guard itself), there is no override mechanism.

<summary>Details</summary>

## Action permissions without enforcement

Nothing in this diff calls `isActionAllowed`. The `processInput` / `processOutput` pipeline in `index.ts` doesn't consult permissions. The API routes that perform write actions (messages, follow-ups, stock updates) don't import or invoke it. An admin can toggle "Update Stok" off in the UI, persist that to the database, and the agent can still update stock because no enforcement boundary exists.

The UI presents these as functional controls. A user toggling permissions off will believe they've restricted the agent. If this is intentional scaffolding, the UI should indicate that enforcement is not yet active.

## Knowledge auditor false-positive risk

The `auditKnowledgeContent` function runs eight regex patterns against knowledge content and hard-rejects on any match. Several patterns are too broad for a customer-service knowledge base:

```typescript
{ pattern: /act\s+as/i, label: "Persona swap: 'act as'" }
{ pattern: /system\s*:/i, label: "System prompt fragment: 'system:'" }
```

The phrase "act as" appears in legitimate Indonesian e-commerce context (product descriptions, role explanations). The pattern "system:" could appear in technical documentation or in entries describing the guard system itself. Since the auditor returns a hard 400 with no override, an admin cannot save entries containing these phrases without modifying code.

The auditor rejects outright rather than strip-and-allow (the input sanitizer's approach). For knowledge content this hard-stop posture makes sense architecturally — you don't want poisoned knowledge — but the broad patterns mean the hard stop will trigger on benign content. Consider requiring more context around trigger phrases (e.g., "ignore.*previous instructions" is already well-scoped; "act as" needs similar anchoring).

## Duplicated zero-width character handling

Both `input-sanitizer.ts` and `knowledge-auditor.ts` define the same regex:

```
/[\u200B\u200C\u200D\uFEFF\u202A-\u202E\u2060\u2066-\u2069]/g
```

If a new Unicode range needs coverage, both files must be updated independently. A shared constant in a common module would reduce that risk.

## `allowedActions` persists arbitrary strings

The PATCH validation in `agent-guard/route.ts`:

```typescript
const parsed = JSON.parse(body.allowedActions);
if (!Array.isArray(parsed) || !parsed.every((item: unknown) => typeof item === "string")) {
  errors.push("allowedActions harus berupa JSON array of strings");
}
```

A client can save `["SEND_MESSAGE", "DELETE_EVERYTHING", "LAUNCH_MISSILES"]` and it persists successfully. The invalid entries are inert (they won't match any real action type), but the database accumulates garbage over time. Importing `ACTION_TYPES` and validating membership would close this.
