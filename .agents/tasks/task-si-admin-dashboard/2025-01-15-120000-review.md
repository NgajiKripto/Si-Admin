# Multi-feature admin dashboard for small business CRM

Si-Admin is a greenfield Next.js 14+ admin dashboard built with Prisma/SQLite, shadcn/ui, and Tailwind CSS. It implements six feature areas from scratch: a dashboard overview, chat management, follow-up task tracking with calendar view, customer feedback with templates, stock/inventory management, and business customization settings. The architecture is a standard Next.js App Router pattern with API route handlers performing Prisma queries and client components fetching via `fetch()`. The application is Indonesian-language throughout and targets small retail businesses.

**Watch for:** Stock movement endpoint allows negative inventory without validation (confirmed). The PATCH endpoints for categories and channels pass raw request body to Prisma `update()` enabling arbitrary field overwrites (confirmed). No authentication or authorization exists on any API route (confirmed). The dashboard page uses `$queryRawUnsafe` with a hardcoded string, but the pattern is still worth noting for future maintenance (confirmed).

## High-level view

The data model is well-structured for a multi-channel CRM, with clear relationships between business profiles, channels, customers, conversations, and inventory. The schema enforces referential integrity through foreign keys, and the seed file provides realistic demo data covering all features.

The API layer follows a consistent pattern: route handlers with try/catch, Prisma queries, and Indonesian-language error messages. The critical gap is that every endpoint is completely unauthenticated and several PATCH routes accept the raw request body without field whitelisting, meaning any Prisma-writable field can be overwritten by a crafted request.

Stock management has a race condition risk in the movement endpoint: reading an item's quantity and updating it via `increment` are separate operations without a transaction, and no guard prevents quantity from going negative. For a small-business tool this might be acceptable, but it will produce nonsensical inventory counts under concurrent use.

The frontend is entirely client-rendered for interactive pages (chat, follow-up, feedback, stock, settings) and server-rendered for the dashboard overview. There is no loading/error boundary strategy beyond inline "Memuat..." text, no optimistic UI, and no real-time updates (the chat uses manual refresh via key increment).

<summary>Issues (7)</summary>

1. **Negative stock quantity** — The stock movement POST endpoint does not check whether an "OUT" movement would drive quantity below zero. Add a pre-check or use a Prisma transaction with a conditional update.
2. **Unprotected PATCH body passthrough** — `PATCH /api/stok/[id]`, `PATCH /api/pengaturan/channels`, and `PATCH /api/pengaturan/categories` forward the raw request body to `prisma.update({ data: body })`. Whitelist allowed fields.
3. **No authentication on any route** — All API routes are publicly accessible. At minimum, add session-based auth middleware or a guard on the API layer before deploying.
4. **Non-transactional stock movement** — Creating a `StockMovement` and updating the `StockItem` quantity are two separate awaited calls. If the second fails, the movement record exists but the quantity is stale. Wrap in `prisma.$transaction`.
5. **Channel deletion without cascade check** — Deleting a channel via `DELETE /api/pengaturan/channels` does not check whether conversations reference that channel. This will throw a foreign-key error at the database level, surfaced as a generic 500.
6. **`$queryRawUnsafe` in dashboard** — The low-stock count uses `$queryRawUnsafe`. While the query is hardcoded (no injection risk today), this will bypass Prisma's type safety and break silently if the schema changes. Replace with a Prisma `count` using a raw `where` filter or a safe raw query.
7. **No input length/size validation** — No API route validates string length, numeric bounds, or payload size. A client can submit arbitrarily large content fields.

## Stock movement integrity

The `POST /api/stok/movement` handler creates a movement record and then increments/decrements the item quantity as two independent operations:

```typescript
const movement = await prisma.stockMovement.create({ ... });

const quantityChange = type === "IN" ? parseInt(quantity) : -parseInt(quantity);
await prisma.stockItem.update({
  where: { id: stockItemId },
  data: {
    quantity: { increment: quantityChange },
    ...(type === "IN" ? { lastRestocked: new Date() } : {}),
  },
});
```

If the update fails (network blip, constraint violation), the movement record persists with no corresponding quantity change. More importantly, nothing prevents `quantity` from going negative. For an inventory system, this is a data integrity concern: a user recording an "OUT" of 100 units when only 5 exist will result in a -95 quantity, and there is no after-the-fact correction mechanism.

The fix is straightforward: wrap both operations in `prisma.$transaction`, and for "OUT" movements, first fetch the current quantity and reject if insufficient.

## Unguarded PATCH endpoints

Three routes accept the request body and pass it directly to Prisma's `update`:

- `PATCH /api/stok/[id]` — `data: body`
- `PATCH /api/pengaturan/channels?id=...` — `data: body`
- `PATCH /api/pengaturan/categories?id=...` — `data: body`

This means a caller can overwrite any scalar field on the model, including `id`, `createdAt`, or relationship fields like `businessId`. In the category case, a malicious request could reassign a category to a different business. The fix is to destructure only the expected fields from the body before passing to Prisma.

## Authentication and authorization posture

There is no authentication layer. No session check, no API key, no middleware guard. Every route is accessible to any HTTP client. The `.env` file contains only the database URL. For a local demo or prototype this is expected, but the code has no TODO markers or commented-out auth placeholders indicating this is a conscious deferral. Before any deployment (even internal), this needs at minimum a session middleware on the `(dashboard)` layout and API route group.

## Channel deletion cascading

The `DELETE /api/pengaturan/channels` handler deletes a channel by ID without checking whether `Conversation` records reference it via `channelId`. The schema defines `channelRef Channel @relation(fields: [channelId], references: [id])` without `onDelete: Cascade`, so SQLite will reject the deletion with a foreign key constraint error. The handler's catch block will return a generic "Gagal menghapus channel" 500 error with no indication of why. Contrast this with the category deletion endpoint, which correctly checks for child items before proceeding.

## Dashboard `$queryRawUnsafe` for low-stock count

The dashboard page constructs the low-stock metric with:

```typescript
prisma.$queryRawUnsafe<{ count: number }[]>(
  'SELECT COUNT(*) as count FROM StockItem WHERE quantity <= minThreshold'
)
```

The preceding `.findMany()` call is dead code (its result is discarded via `.then(() => ...)`). The raw query itself is safe from injection since there are no parameters, but `$queryRawUnsafe` is the wrong tool when the query is static. Using `$queryRaw` (tagged template) would be more appropriate, and the dead `findMany` call should be removed to avoid confusion and wasted database round-trips.

## Client-side data fetching patterns

All interactive pages fetch data in `useEffect` without an abort controller or stale-closure guard. In the `ConversationList` component, `fetchConversations` runs on every change to `search` or `channelFilter`. Rapid typing will fire multiple concurrent requests, and depending on network timing, an older response may arrive after a newer one and overwrite the state with stale data. The standard fix is an `AbortController` in the effect cleanup, or a request-ID check in the response handler.

The `MessageView` component has a React-specific issue: the `useEffect` that calls `fetchMessages` depends on `conversationId` but the function `fetchMessages` is defined outside the effect and captures the current `conversationId` via closure. This works because the effect re-runs when `conversationId` changes, but the ESLint exhaustive-deps rule would flag `fetchMessages` as a missing dependency.

## Test coverage

There are no test files in this diff. No unit tests for API routes, no integration tests for the Prisma layer, no component tests for the React pages. The seed file provides demo data but is not a substitute for automated testing. The most critical untested paths are:

- Stock movement negative-quantity edge case
- Concurrent movement requests (race condition)
- Category/channel deletion with dependent records
- Chat message creation with invalid `conversationId`
