# Knowledge Base and Agent Intelligence system for Si-Admin

This change adds two major feature areas to the customer service admin dashboard: a Knowledge Base (Basis Pengetahuan) with full CRUD for training data, and an Agent Intelligence page (Kecerdasan Agent) providing a 4-tier memory system, self-improvement tracking, knowledge graph, hybrid BM25 search, and a health dashboard. Five new Prisma models back the features, with API routes following the existing pattern. The README was also redesigned into a professional format with badges, architecture diagrams, and API documentation.

**Watch for:** Memory consolidation never deletes source records, causing unbounded growth (confirmed). The hybrid search loads all records into memory for scoring (confirmed). Bulk import fires sequential unbatched HTTP requests with no transaction boundary (confirmed). The `page` and `limit` query params are parsed with `parseInt` without bounds validation, accepting negative values or NaN (confirmed).

---

## High-level view

The Prisma schema adds five models (KnowledgeEntry, AgentMemory, LearningInsight, KnowledgeGraphEntity, KnowledgeGraphRelation) using string-typed enums rather than Prisma's native enum, consistent with the SQLite provider constraint. The models are standalone -- no foreign keys link them to the existing Customer/Conversation models, which keeps migration simple but means the "intelligence" layer operates in its own silo.

The API layer follows the existing project pattern: RESTful routes in `src/app/api/`, simple Prisma queries, error responses with Indonesian messages. Input validation exists but is shallow -- required field checks and enum allowlists, no length limits, no sanitization of JSON string fields.

The memory consolidation endpoint implements a time-based promotion scheme (Working -> Episodic -> Semantic -> Procedural) that creates new records without removing originals. Working memories have their strength lowered to 0.3 but remain in the table indefinitely, and episodic-to-semantic promotion creates one new record per old episodic memory without any cleanup.

The hybrid search endpoint loads all active knowledge entries and all agent memories into application memory, then scores them with a BM25-like formula. This is fine for seed-scale data but becomes a concern if the knowledge base grows into thousands of entries.

The Knowledge Base UI is a straightforward CRUD interface with a test panel that simulates agent responses using keyword matching. The bulk import feature parses text blocks separated by double-newline or `---` and fires individual POST requests sequentially, meaning a 100-block import sends 100 HTTP requests with no atomicity guarantee.

<summary>Issues (7)</summary>

1. **Unbounded memory growth from consolidation** — Consolidation creates new tier-promoted records but never deletes or archives the source records. Add a cleanup step or archive flag after successful promotion.
2. **Full-table scan in search** — `POST /api/agent/search` fetches all knowledge entries and all memories into Node.js memory. Add database-level text filtering (WHERE contains) before in-memory scoring, or paginate the source data.
3. **Bulk import has no atomicity** — Each entry is created via a separate HTTP POST. A failure partway through leaves partial data. Use a batch creation endpoint or wrap in a Prisma transaction.
4. **No input length validation on text fields** — API routes accept arbitrarily large `content` strings. A single POST could insert megabytes into SQLite. Add max-length checks on content, title, and metadata fields.
5. **parseInt without bounds on pagination params** — `page` and `limit` in `/api/knowledge` accept negative numbers or NaN (falls through to undefined behavior in Prisma's `skip`/`take`). Clamp to positive integers with sensible maximums.
6. **Graph entity properties stored as unvalidated string** — The `properties` field accepts arbitrary strings meant to be JSON but is never parsed or validated server-side. Malformed JSON will be stored and potentially break client-side `JSON.parse`.
7. **Knowledge test loads entire table** — `POST /api/knowledge/test` fetches all active entries for keyword scoring. Same scaling concern as hybrid search.

## Memory consolidation grows without bound

The consolidation endpoint at `/api/agent/memory/consolidate` implements three promotion paths:

- Working memories older than 1 hour get summarized into a single episodic entry, then have their `strength` reduced to 0.3.
- Episodic memories older than 24 hours each spawn a new semantic entry.
- Semantic memories with `accessCount >= 3` spawn a procedural entry (with a duplicate check via `metadata contains`).

The working-to-episodic path concatenates all old working memories into a single string, truncated at 500 characters. If there are many working memories, most content is silently dropped. More critically, none of the three paths delete or mark the source records as consumed. Repeated consolidation runs will re-process the same episodic memories (since they remain older than 24 hours) and create duplicate semantic entries each time.

The duplicate guard for semantic-to-procedural (checking `metadata contains mem.id`) is fragile -- it depends on the source ID appearing as a substring in any procedural memory's metadata JSON string, which will fail if the metadata format changes.

## Search scalability

```typescript
const [knowledgeEntries, memories] = await Promise.all([
  prisma.knowledgeEntry.findMany({ where: { isActive: true } }),
  prisma.agentMemory.findMany(),
]);
```

Both `/api/agent/search` and `/api/knowledge/test` load their respective full tables into memory before scoring. The BM25 implementation itself is well-structured (saturating TF, proper IDF), but the approach only works at small scale. With SQLite, a pragmatic improvement would be filtering to entries whose content contains at least one keyword before loading.

## Bulk import lacks transaction safety

`BulkImport.tsx` iterates over parsed text blocks and fires one `fetch("/api/knowledge", { method: "POST" })` per block. If the 50th request fails, the first 49 entries remain in the database. The user sees an alert with the count but has no way to roll back. A single batch endpoint using `prisma.knowledgeEntry.createMany` would be both faster and atomic.

## Input validation gaps across API routes

The POST handlers validate required fields and enum membership but apply no length constraints. Relevant endpoints:

- `/api/knowledge` POST: `content` can be any string length.
- `/api/agent/memory` POST: `content` and `metadata` unchecked.
- `/api/agent/graph` POST: `properties` accepted as-is, intended to be JSON but never validated.
- `/api/agent/learning` POST: `content` unbounded.

The `GET /api/knowledge` route parses `limit` via `parseInt(searchParams.get("limit") || "20")`. If a caller passes `limit=-1`, Prisma's `take: -1` may behave unexpectedly. Similarly, `page=0` produces `skip: -20`.

## Consolidation's "applied" flag has an asymmetry

The learning insight `PUT /api/agent/learning/[id]` allows setting `applied: true` (which also sets `appliedAt` to now) but if `applied: false` is sent, `appliedAt` is not cleared. An insight that was "un-applied" retains its original `appliedAt` timestamp, giving a misleading data trail.

```typescript
data: {
  ...(applied !== undefined ? { applied } : {}),
  ...(applied ? { appliedAt: new Date() } : {}),  // only sets on true
  ...(impact !== undefined ? { impact } : {}),
}
```

## Knowledge graph GET fetches redundant data

The `/api/agent/graph` GET handler fetches entities with their nested `fromRelations` and `toRelations` (each including the related entity), then separately fetches all relations with their entities. This returns the same relation data twice in different shapes, roughly tripling the response payload for no additional information.
