# Knowledge Base and Agent Intelligence system (v2)

This change adds a Knowledge Base and Agent Intelligence system to the Si-Admin customer service dashboard: full CRUD for training data with bulk import, a 4-tier memory system with time-based consolidation, a learning insights tracker, knowledge graph management, hybrid BM25 search, and a health metrics dashboard. Five Prisma models back the features, with 10 API route files and 10 UI components.

All seven v1 issues and the two additional issues have been addressed. Three new concerns remain, none blocking.

**Watch for:** Consolidation lacks transactional safety -- if a delete fails after creates succeed, data is lost without recovery (likely). The `question` field on `/api/knowledge/test` has no length limit, allowing arbitrarily large OR clause generation (confirmed). The `metadata` field on relations endpoint has no length validation (confirmed).

---

## High-level view

The consolidation endpoint now deletes source records after promotion, solving the unbounded growth problem. However, each promotion path performs creates and deletes as separate operations without a wrapping `$transaction`. A failure between creation and deletion leaves the system in an inconsistent state -- either data loss or duplication on the next run. The risk is low for SQLite in a single-server scenario but would matter if this migrates to a networked database.

The search and test endpoints now use `OR` conditions built from tokenized keywords, filtering at the database level and capping results at 100 rows before in-memory scoring. One edge remains: the `question` and `query` inputs have no length validation, so a sufficiently long input generates a proportionally large OR clause (3 conditions per keyword). Every other endpoint validates input length; these two don't.

Input validation is consistent across write endpoints: `title` capped at 500 chars, `content` at 50,000, `metadata` at 10,000, `properties` parsed as JSON. The graph relations endpoint is the one gap -- it validates entity existence but applies no length check on `metadata`.

## Consolidation atomicity gap

The structure of each promotion path is:

```
1. Query source records
2. Create promoted record(s)
3. deleteMany on source IDs
```

If step 3 fails (e.g., a constraint violation from a concurrent request, or an unexpected runtime error), the newly created promoted records persist and the sources also persist, leading to duplication on the next consolidation run. If a future refactor reorders operations, records could be deleted before the create succeeds, causing data loss.

Wrapping each path in `prisma.$transaction([...creates, deleteMany])` would make each promotion atomic. The batch endpoint already demonstrates this pattern. For the semantic-to-procedural path, the sequential `findFirst` duplicate check inside the loop means a simple `$transaction` array won't work directly -- the logic would need restructuring to collect all creates first, then execute them with the delete in one transaction.

## Input length on search endpoints

Both search endpoints tokenize the input then build an OR clause with conditions per keyword:

```typescript
// /api/knowledge/test: 3 conditions per keyword (title, content, tags)
const orConditions = keywords.flatMap((keyword) => [
  { title: { contains: keyword } },
  { content: { contains: keyword } },
  { tags: { contains: keyword } },
]);
```

A 500-word input produces ~500 keywords (after the `length > 2` filter) and thus 1500 OR conditions for the test endpoint. A simple `if (question.length > 1000)` guard would close the gap with the other validated endpoints.

## Relations metadata validation

The entity POST endpoint validates `properties` for both length (10,000 chars) and JSON validity. The relations POST endpoint accepts `metadata` and passes it directly to Prisma without any check:

```typescript
metadata: metadata || null, // no length or format check
```

The consolidation endpoint uses `metadata` to store source IDs (via `contains` queries), so malformed or oversized metadata could degrade those lookup queries over time.
