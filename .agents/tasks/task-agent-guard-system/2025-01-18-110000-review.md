# Multi-layer agent guard for Indonesian customer service (v2)

This change adds an input/output guardrail pipeline for the customer service AI agent: prompt injection detection, keyword-based scope classification, response truncation, and output validation (PII redaction, blocked pattern removal, system prompt leak detection). Configuration is persisted in a Prisma model and managed via an admin UI tab. The seven issues raised in v1 have all been addressed: the JSDoc warning on `sanitizeInput`, custom topic keyword fallback, relative confidence formula, token-before-character truncation ordering, API validation, consistent PII detection on `sanitizedOutput`, and error state in the UI.

Watch for: the "structured" format path still wraps truncated text in a JSON object regardless of whether the original response was JSON, producing double-encoding in the common case (confirmed); the confidence formula still allows single generic keywords like "ada" to clear the 0.3 threshold on smaller keyword lists (confirmed, minor); client-side `parseInt || 0` can send out-of-bounds values that the API rejects with a 400, but the UX is confusing since the input field shows the invalid value until refresh (likely).

**Verdict**: APPROVED

## High-level view

Custom topics now fall back to `[topic]` as keywords, meaning an admin-added "asuransi" topic will match messages containing the word "asuransi" without needing a hardcoded keyword list.

The confidence formula change from `matchCount / (words.length * 0.5)` to `matchCount / Math.max(3, keywords.length * 0.3)` eliminates the input-length-dependent inflation. The formula is still generous—one keyword match on a 9-item list clears the 0.3 threshold—but this is a usable scope tagger rather than the broken one from v1.

The structured format JSON wrapping still applies after truncation, which means valid LLM JSON that gets truncated is wrapped in `{"message":"..."}`. Detecting "was this originally JSON" adds complexity, and the wrapping at least produces parseable output, but downstream consumers need to handle the double-encoding case.

The UI number inputs use `parseInt || 0` which can produce out-of-range values. The API now correctly rejects these with a 400 and the error banner displays, but the input field retains the invalid value rather than reverting.

<summary>Issues (3)</summary>

1. **Structured format double-encoding** — When the LLM returns valid JSON that exceeds length limits, truncation invalidates it, and the catch block wraps the truncated string (including "...") in `{"message":"..."}`. The consumer receives double-encoded JSON. Consider detecting JSON input before truncation and truncating at a structural boundary, or documenting that "structured" format requires the LLM output to already be within limits.

2. **Single-keyword threshold clearing** — With `Math.max(3, keywords.length * 0.3)` as denominator, a single keyword match on any topic with fewer than 10 keywords yields confidence >= 0.33, clearing the 0.3 threshold. The word "ada" matches `stok`, "buka" matches `jam_operasional`. The classifier rarely rejects anything with common Indonesian vocabulary in the input.

3. **Client-side parseInt fallback mismatch** — The number inputs use `parseInt(e.target.value) || 0`. If a user clears the field or types "abc", the state becomes 0. The API rejects 0 (below minimum 50/10), returning a 400. The user sees the error banner but the input still shows "0" and requires manual correction. Using `Math.max(min, parseInt(...) || min)` in the onChange handler would prevent this disconnect.

## Structured format truncation path

The structured-format block tries `JSON.parse(content)` on already-truncated content and on failure wraps in `{"message": content}`:

```typescript
if (format === "structured") {
  try {
    JSON.parse(content);
  } catch {
    content = JSON.stringify({ message: content });
  }
}
```

If the LLM returned `{"products": [{"name": "Widget", "price": 50000}, ...]}` and this gets truncated to `{"products": [{"name": "Widget", "pri...`, the consumer receives `{"message":"{\"products\": [{\"name\": \"Widget\", \"pri..."}`. The "..." indicator lives inside the JSON string value. A downstream JSON consumer would need to know to extract `.message` and understand it's a truncated fragment, not a real message. This isn't a regression from v1—the ordering fix (token before character) was the critical change—but the structured path remains a source of surprising output for API consumers.

## Scope classifier confidence at the margins

The new formula normalizes against keyword list size:

```
denominator = Math.max(3, keywords.length * 0.3)
```

For `stok` (9 keywords): denominator = `Math.max(3, 2.7)` = 3. One match = 0.33, clearing the 0.3 threshold. The word "ada" (meaning "is there" / "exists") is extremely common in Indonesian and appears in the `stok` keyword list. A message like "ada yang bisa bantu?" (can anyone help?) would be classified as in-scope under `stok`. The scope gate functions as a topic tagger rather than a hard gate. If the guard is meant to block genuinely off-topic requests (politics, tech support for unrelated products), the common words in keyword lists would need removal.

## UI number input and API validation interplay

The frontend sends raw `parseInt(e.target.value) || 0` values. The API validates `val < 50` for `maxResponseLength` and `val < 10` for `maxResponseTokens`, returning 400 with Indonesian error messages. The error banner displays correctly. The disconnect is that the user's input field retains "0" after the failed save—it doesn't revert to the last-saved value or enforce bounds on change. The `min`/`max` HTML attributes on the `<Input>` only affect the browser's built-in validation (which is bypassed since there's no `<form>` submission), so they serve as hints rather than enforcement.
