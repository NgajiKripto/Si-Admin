# Security hardening and code quality sweep for Si-Admin

This change applies 14 fixes across the Si-Admin application's API surface: centralized auth middleware for all admin routes, hardcoded secret removal, input validation, race condition fix in stock updates, rate limiting for chat endpoints, shared session creation, word-boundary fix in the scope classifier, pagination, JSON.parse safety, and a test suite for the agent-guard modules. The approach is broad but shallow — each fix is localized and mechanical, which keeps the blast radius small but also means some fixes don't go deep enough to close the gap they intend to address.

**Watch for:** The OpenAI API key validation still returns an empty string in non-production, so dev callers silently get an invalid key that will fail at the OpenAI API boundary rather than at config time (confirmed). The rate limiter is a module-level singleton and will not survive serverless cold starts or multi-instance deployments — it's an in-process data structure presented as production protection (confirmed). The chat endpoints have no auth at all, relying only on rate limiting by session ID — an attacker can fabricate session IDs to bypass limits (confirmed).

## High-level view

Auth is now centralized in `src/lib/auth.ts` and guarded against missing secrets in production via a throw-at-module-load pattern. All admin CRUD routes import and apply `requireAuth`, but the two chat endpoints (`/api/agent/chat` and `/api/agent/chat/stream`) are intentionally excluded, creating a two-tier access model where the chat surface is protected only by rate limiting and message length caps.

The stock update tool's race condition fix moves findUnique, quantity validation, and the write all inside a single Prisma `$transaction` callback, eliminating the TOCTOU gap.

The rate limiter uses a sliding-window timestamp array stored in a plain `Map`. It works in a single-process dev server but provides no protection in serverless or horizontally-scaled deployments.

The scope classifier now uses `\b` word-boundary regex instead of `includes()`, fixing false positives for Indonesian words that share prefixes (e.g., "produk" vs "produksi"). Note that `\b` operates on ASCII word boundaries — if the keyword list ever includes non-Latin characters, matches will break silently.

The agent-guard runner functions are now fully async, using `spawn` with stdin piping and `promisify(execFile)` instead of `execSync`. The existing in-process callers (`processInput`/`processOutput`) use the TypeScript implementations directly and are unaffected by this signature change.

1. **OpenAI key still empty-string in dev** — `getOpenAIConfig()` warns but returns `apiKey: ""`, which will fail at the OpenAI HTTP boundary. Throw or return a clearly invalid sentinel that forces callers to handle it.
2. **Rate limiter ineffective in serverless/multi-instance** — The in-memory `Map` resets on every cold start and is not shared between instances. Add a comment documenting this limitation, or use an external store (Redis, Upstash) for production.
3. **Chat endpoints unauthenticated with bypassable rate-limit key** — Rate limiting keys off `sessionId` from the request body, which the client controls. An attacker can rotate session IDs to get unlimited requests. Key off a server-side identity (IP + fingerprint, or require auth).
4. **Stock API response shape is a breaking change** — GET previously returned a flat array; now returns `{items, total, page, limit, totalPages}`. Frontend consumers of this endpoint will break unless updated simultaneously.
5. **Auth module-load throw may crash unrelated imports** — The `throw` at module top level in `auth.ts` executes when the module is first imported. In production if `ADMIN_SECRET` is missing, the entire process crashes during module resolution rather than at request time. This is arguably correct (fail fast) but should be documented.
6. **`getOrCreateSession` creates orphan sessions on concurrent requests** — Two requests with the same `sessionId` can both find no existing session and both create new sessions. The second `create` will succeed (CUID IDs are unique), resulting in orphaned sessions.

## Unauthenticated chat surface and rate-limit bypass

The chat endpoints (`/api/agent/chat` and `/api/agent/chat/stream`) deliberately skip `requireAuth`. The rate limiter uses a key derived from the request body's `sessionId` field:

```typescript
const rateLimitKey = sessionId || request.headers.get("x-forwarded-for") || "anonymous";
```

A client that omits `sessionId` falls back to `x-forwarded-for`, which is trivially spoofable via proxy headers. A client that provides a fresh random `sessionId` on every request gets a fresh rate-limit bucket. The rate limiter offers no meaningful protection against a motivated attacker — it only throttles well-behaved clients.

If the intent is to keep chat open for unauthenticated end users, the rate limit key should be derived from something the server controls (client IP from the socket, or a server-issued opaque token).

## OpenAI config gap in development

The new validation in `getOpenAIConfig()` only throws in production:

```typescript
const apiKey = process.env.OPENAI_API_KEY || "";
if (!apiKey && process.env.NODE_ENV === "production") {
  throw new Error("OPENAI_API_KEY environment variable is required");
}
if (!apiKey) {
  console.warn("[langchain] OPENAI_API_KEY not set, API calls will fail");
}
```

In development, it returns `apiKey: ""`. Any code path that actually calls OpenAI (and the chat endpoints do) will get a cryptic HTTP 401 from OpenAI rather than a clear local error. The original issue flagged "empty string fallback" as a problem — this fix addresses production but leaves the same failure mode in dev, just with a warning.

## Rate limiter memory characteristics

For a serverless deployment (Vercel), the `setInterval` cleanup never fires because the process lifetime is typically shorter than 5 minutes — meaning no cleanup ever runs, and the limiter resets on every cold start anyway. The implementation is effectively a dev-mode throttle presented as production security infrastructure.

## Stock API pagination as a breaking change

The GET `/api/stok` endpoint previously returned a flat JSON array of stock items. It now returns:

```json
{ "items": [...], "total": 42, "page": 1, "limit": 20, "totalPages": 3 }
```

Any frontend component that does `const items = await fetch('/api/stok').then(r => r.json())` and iterates the result directly will break. This is a backwards-incompatible API change that needs coordinated frontend updates.

## Test coverage gaps

The 25 tests cover the pure-function modules of agent-guard: `sanitizeInput`, `classifyScope`, `validateOutput`, and `isActionAllowed`.

Not tested: the rate limiter, the auth middleware, the session helper, the stock update tool's transactional behavior, or any integration between components. The runner.ts async conversion has no tests (the tests call the TypeScript implementations directly, not the runner wrappers). The pagination logic in the stock API is untested.
