# Bold visual redesign: typography, color, animation, bento layout

The dashboard gets a complete visual overhaul: Inter replaced with Space Grotesk + DM Sans, the neutral grayscale swapped for a deep navy/teal/amber oklch palette, CSS keyframe animations added for entrance effects, and the metric cards reorganized from a uniform 4-column grid into an asymmetric bento layout with a 2x2 hero card. The sidebar gains texture (noise overlay, gradient background, diagonal accent stripe) and hover micro-interactions. All existing Prisma queries, navigation, and Indonesian labels remain unchanged.

**Watch for:** StaggerReveal wrapping each card in a `<div>` breaks the bento grid's `col-span`/`row-span` classes (confirmed). No `prefers-reduced-motion` handling means users who need reduced motion see invisible content (confirmed). The self-referencing `--font-display: var(--font-display)` in the `@theme inline` block may fail to resolve correctly (likely). A raw oklch value in a Tailwind class without square brackets won't generate valid CSS (confirmed).

## High-level view

The font swap from Inter to Space Grotesk (display) and DM Sans (body) follows a clean pattern through `next/font/google` CSS variables, but the `@theme inline` block defines `--font-display: var(--font-display)` which is a circular reference — the font utility classes (`font-display`, `font-body`) might silently fall back to the browser default.

The color palette commits fully to a dark-first design: `:root` now uses deep navies (oklch 0.13) rather than whites, and the `.dark` variant is nearly identical. The app is effectively dark-mode-only with no usable light theme, which aligns with the stated design intent but is a one-way door.

The bento grid layout relies on CSS grid `col-span-2` and `row-span-2` for asymmetric card sizes, but `StaggerReveal` wraps each child in a plain `<div>` that doesn't forward grid placement classes. The hero card's spanning behavior won't work as intended.

The animation system starts elements at `opacity: 0` and fades them in. Without `prefers-reduced-motion` handling, content is invisible until the animation completes — a functional accessibility gap, not just a preference issue.

## StaggerReveal grid-item absorption

`StaggerReveal` iterates over `React.Children.toArray(children)` and wraps each child in:

```tsx
<div className={cn("animate-stagger-fade-in", className)} style={...}>
  {child}
</div>
```

The dashboard page puts this inside a CSS grid container (`grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-4`). The hero `Card` carries `sm:col-span-2 lg:col-span-2 lg:row-span-2`, but those classes are on the `Card` element which is now a grandchild of the grid — the grid only sees the wrapper `<div>`. The wrapper has no spanning classes, so every card occupies a single cell.

Fix options: pass grid-placement classes through to the wrapper (via a prop or by cloning classes from the child), use CSS `:nth-child` selectors for stagger delays instead of a wrapper component, or make `StaggerReveal` render with `display: contents` on the wrapper divs (which preserves grid participation but has quirks with animation targets).

## Accessibility: motion and contrast

No `@media (prefers-reduced-motion: reduce)` rule exists in `globals.css`. The `animate-stagger-fade-in` class starts at `opacity: 0` and animates to `opacity: 1` — for users with reduced-motion preferences, content will remain invisible until the animation completes (or indefinitely if the browser skips the animation). The fix:

```css
@media (prefers-reduced-motion: reduce) {
  .animate-stagger-fade-in,
  .animate-slide-in-left {
    animation: none;
    opacity: 1;
  }
}
```

The `text-card-foreground/80` class on card titles applies 80% alpha to `oklch(0.95 0.01 260)` over `oklch(0.18 0.02 260)` backgrounds. This may dip below 4.5:1 contrast for the `text-sm` card titles depending on rendering.

## Font variable circular reference

In the `@theme inline` block:

```css
--font-display: var(--font-display);
--font-body: var(--font-body);
```

The intent is to wire next/font's CSS custom properties into Tailwind's theme so that `font-display` and `font-body` utility classes work. In Tailwind v4, `@theme inline` entries define design tokens — when `--font-display` resolves, it references itself, and the browser treats this as an invalid value (initial). Whether Tailwind's compiler handles this as a special case at build time or whether it breaks at runtime depends on the resolution order between next/font's injected `<style>` tag and Tailwind's compiled output.

## Invalid arbitrary value syntax in sidebar gradient

```tsx
<div className="absolute inset-0 bg-gradient-to-b from-sidebar via-sidebar to-oklch(0.75_0.18_180_/_0.05) pointer-events-none" />
```

Tailwind v4 requires square brackets for arbitrary values: `to-[oklch(0.75_0.18_180_/_0.05)]`. Without brackets, this is not a valid utility class and Tailwind will not generate any CSS for it. The gradient will render as a two-stop from-sidebar/via-sidebar gradient with no teal endpoint.
