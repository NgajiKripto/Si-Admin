# LangChain agent upgrade fixes (v2)

All nine issues from the v1 review have been addressed. The streaming endpoint no longer double-invokes the graph, vector search is capped at 1000 records, the approval endpoint requires an `x-admin-token` header, trace data is truncated at 100KB, recursion limits are wired to all graph invocations, guarded tools enforce `allowedActions` inside sub-agents, and memory consolidation archives rather than deletes source records. Documentation comments cover the fire-and-forget metrics trade-off and the LLM singleton lifecycle.

**Watch for:** Trace truncation can produce invalid JSON that downstream consumers silently fail on (confirmed). The approval-resume path in `/api/agent/chat` bypasses both `x-admin-token` auth and `allowedActions` guards (confirmed). The single-agent graph path still uses unguarded tools (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The trace truncation fix applies a two-pass strategy: first halve the trace array, then hard-cut the string to 100KB. The second pass (`substring(0, MAX_TRACE_SIZE)`) will almost certainly produce malformed JSON, which the traces endpoint later tries to `JSON.parse`. The stored data would become unrecoverable garbage rather than a gracefully degraded trace.

The `x-admin-token` auth on the approval endpoint closes the v1 vulnerability, but the approval-resume flow in `/api/agent/chat` (the `approvalId` branch) still executes tool calls with no auth check and using raw `getAgentTools()` without `createGuardedTools()`. An attacker who obtains an approved queue ID can trigger tool execution through the chat endpoint instead.

<summary>Issues (4)</summary>

1. **Trace substring produces invalid JSON** — When `traceJson.length > MAX_TRACE_SIZE` after the array-halving pass, `traceJson.substring(0, MAX_TRACE_SIZE)` creates truncated (unparseable) JSON. The traces endpoint will return `[]` or throw. Either trim the trace array iteratively until it fits, or wrap the stored value in a known-parseable envelope indicating truncation.
2. **Approval-resume in chat route bypasses auth and guards** — `POST /api/agent/chat` with `approvalId` executes `getAgentTools()` directly (no `createGuardedTools`, no `x-admin-token` check). Add auth and guard checks to this code path, or route approval execution exclusively through the now-protected `/api/agent/approval` endpoint.
3. **Single-agent graph uses unguarded tools** — `execute-tool.ts` creates a module-level `ToolNode(getAgentTools())` without `createGuardedTools()`. If the single-agent graph is used (via `getGraph("single")`), `allowedActions` restrictions are not enforced. Either remove the single-agent path or wire guarded tools into it.
4. **Archived tier values are freeform strings** — The archive fix uses `"WORKING_ARCHIVED"`, `"EPISODIC_ARCHIVED"`, `"SEMANTIC_ARCHIVED"` as tier values, but the Prisma schema has no enum and the memory route filters with `{ not: { contains: "_ARCHIVED" } }`. A memory with a tier like `"CUSTOM_ARCHIVED_TEST"` would also be filtered, and there's no validation that only known tier values exist. Low risk for now, but a string enum or validation would prevent drift.

## Trace truncation produces invalid JSON

In `tracing-handler.ts`, the truncation logic has two passes:

```typescript
if (traceJson.length > MAX_TRACE_SIZE) {
  const truncated = this.trace.slice(-Math.floor(this.trace.length / 2));
  traceJson = JSON.stringify(truncated);
  if (traceJson.length > MAX_TRACE_SIZE) {
    traceJson = traceJson.substring(0, MAX_TRACE_SIZE);
  }
}
```

The first pass (halving the array and re-serializing) is sound. The second pass is the problem: if the halved array still serializes to >100KB, the raw `substring` breaks JSON structure. For a long agent session with many tool calls producing verbose output, this is reachable. The traces endpoint (`/api/agent/metrics/traces`) calls `JSON.parse(record.errorMessage)` on retrieval, which will throw on the corrupted string, and the `catch` block returns an empty array with no indication that data was lost.

Fix: replace the `substring` fallback with an iterative approach that keeps removing array elements until the serialized length fits.

## Approval-resume path unguarded

In `src/app/api/agent/chat/route.ts`, lines 20-62 handle the `approvalId` branch:

```typescript
if (approvalId) {
  const queueItem = await prisma.humanApprovalQueue.findUnique({ ... });
  if (queueItem.status === "APPROVED") {
    const tools = getAgentTools();
    const tool = tools.find((t) => t.name === payload.toolName);
    const result = await tool.invoke(payload.args);
    // ...
  }
}
```

This path has no `checkAuth()` or equivalent, and uses `getAgentTools()` (unguarded) rather than `createGuardedTools()`. The approval endpoint itself is now properly authenticated, but the same execution logic is duplicated in the chat route without protection. An attacker who learns an approved approval ID can call the chat endpoint to execute the tool without presenting `x-admin-token`.

The simplest fix is to remove the approval-resume logic from the chat route entirely and direct clients to use the authenticated approval endpoint. Alternatively, add the same auth check here.

## Single-agent path does not enforce action permissions

`src/lib/langchain/graph/nodes/execute-tool.ts` declares a module-level `ToolNode`:

```typescript
const toolNode = new ToolNode(getAgentTools());
```

This is used by the single-agent graph compiled in `src/lib/langchain/graph/index.ts`. The multi-agent path correctly passes `guardConfig` down to sub-agents which call `createGuardedTools()`, but the single-agent path has no equivalent. If a developer or future feature uses `getGraph("single")`, all tools execute without permission checks.

Lower risk because the current code defaults to `"multi"` everywhere, but the single-agent graph is still exported and callable.
