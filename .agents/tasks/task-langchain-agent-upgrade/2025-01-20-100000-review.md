# LangChain agent upgrade with multi-agent routing, hybrid search, and observability

Si-Admin gains a full LangChain.js / LangGraph.js agent layer: an LLM router dispatches to CS, Stock, and Follow-up sub-agents, each with their own tool set. Context retrieval uses BM25 with optional vector-similarity fusion (RRF). Memory consolidation (working -> episodic -> semantic -> procedural) is now LLM-powered. Observability lands via callback handlers that write per-request metrics and traces to the database. An SSE streaming endpoint and human-in-the-loop approval queue complete the surface area.

**Watch for:** The streaming endpoint invokes the graph twice per request (once to stream, once to get the final response), doubling LLM cost and creating duplicate side-effects (confirmed). Vector search loads all embeddings into memory on every query, which will collapse under real data volumes (confirmed). The approval execution path in `/api/agent/approval` has no authorization and executes arbitrary tool calls from JSON payloads (confirmed). Trace data is stored in the `errorMessage` column of `AgentMetrics`, conflating two unrelated concerns and limiting trace size to whatever SQLite allows in a text column (confirmed).

## High-level view

The multi-agent graph wraps each domain (CS, stock, follow-up) as an independent LangGraph `StateGraph` compiled sub-graph. An LLM-based router classifies incoming messages into one of three categories, then hands off to the corresponding sub-agent. Input and output guards bracket the entire flow, so the existing Agent Guard system applies to all routes. The single-agent graph (with retrieve_context and tool execution) remains available but defaults to multi-agent in all API endpoints.

Vector search is conceptually hybrid but architecturally naive: `searchByVector` loads the entire `VectorEmbedding` table into memory, parses every stored JSON embedding, and computes cosine similarity in a loop. This is fine for tens of entries during development but becomes a hard blocker at scale. The BM25 path is solid and handles the fallback when no embeddings exist.

The SSE streaming implementation streams partial word tokens during the first graph invocation but then calls `graph.invoke` a second time to get `finalResponse`, which re-runs the entire agent pipeline. This means every streamed request produces two full LLM call chains, two sets of tool executions, and two database metric writes.

The approval system gates high-impact actions (stock updates > 50, high-priority follow-ups) behind a queue, but the POST endpoint that resolves approvals has no authentication. Any caller who can guess or enumerate queue IDs can approve actions and trigger tool execution.

Observability stores everything in a single `AgentMetrics` table. Traces are serialized into the `errorMessage` field, which is a semantic mismatch. The `MetricsHandler` fires database writes from callback handlers without awaiting them, so metric loss is possible under load or if the process exits during a request.

<details>
<summary>Issues (9)</summary>

1. **Double graph invocation in SSE stream** — The streaming endpoint calls `graph.stream()` then `graph.invoke()` on the same input, doubling LLM calls, tool side-effects, and cost. Remove the second `invoke` and extract `finalResponse` from the streamed state instead.
2. **Unbounded vector scan** — `searchByVector` loads all `VectorEmbedding` rows into memory and computes similarity in JS. Add pagination, pre-filtering, or move to a vector-capable store before the table grows.
3. **Unauthenticated approval execution** — `POST /api/agent/approval` executes tool calls upon approval with no auth check. Add authorization (at minimum, verify caller identity matches `resolvedBy`).
4. **Trace stored in errorMessage column** — `TracingHandler.saveTrace()` writes serialized trace JSON into `AgentMetrics.errorMessage`. Use a dedicated column or table to avoid confusing traces with actual errors and to prevent silent truncation.
5. **Sub-agents lack iteration caps** — CS, stock, and follow-up sub-graphs loop `llm -> tools -> llm` with no maximum iteration count. A pathological tool-call loop will run until timeout or token exhaustion. Wire `AGENT_MAX_ITERATIONS` to LangGraph's `recursionLimit` config.
6. **Fire-and-forget metric writes may lose data** — `MetricsHandler` uses `.catch()` on Prisma writes without retry. Under high concurrency or process termination, metrics will be silently lost. Consider batching writes or awaiting them before response.
7. **LLM singleton not safe for config changes** — `getLLM()` creates a module-level singleton. If environment variables change (e.g., API key rotation), the stale instance persists until process restart.
8. **allowedActions guard not enforced in multi-agent path** — `guardConfig.allowedActions` is loaded but never checked before sub-agents execute tools. Stock updates and follow-up creation proceed regardless of the guard's action permissions. Enforce the allow-list before tool invocation.
9. **Memory consolidation hard-deletes source records** — Promoted memories are deleted with no soft-delete or archive. If LLM summarization produces a poor result, original data is irrecoverable. Add a soft-delete flag or archive table.

## Double invocation in the streaming endpoint

In `src/app/api/agent/chat/stream/route.ts`, the `ReadableStream.start` callback first iterates over `graph.stream(...)` (lines 76-107), emitting word-by-word tokens from any node that produces a `finalResponse`. After that loop completes, it calls `graph.invoke(...)` again (lines 110-116) with the identical input and same session to get `finalResponse` for the `done` event. This second invocation re-runs the full agent pipeline: guard, router, sub-agent LLM calls, tool executions, and all callback handlers.

```typescript
// First pass: streams partial results
const eventStream = await graph.stream(
  { messages: [new HumanMessage(message)], sessionId: session.id },
  { streamMode: "updates", callbacks }
);
for await (const event of eventStream) { /* emit tokens */ }

// Second pass: full re-execution for finalResponse
const result = await graph.invoke(
  { messages: [new HumanMessage(message)], sessionId: session.id },
  { callbacks }
);
```

The `graph.stream()` call already produces node outputs including `finalResponse`. The second `invoke` should be removed; instead, accumulate `finalResponse` from the streamed events.

## Vector search memory pressure

`searchByVector` in `src/lib/langchain/embeddings-service.ts` calls `prisma.vectorEmbedding.findMany({ where })` with no `take` limit, deserializes every embedding JSON string into a `number[]`, then iterates to compute cosine similarity. For `text-embedding-3-small` with 1536 dimensions, each record is roughly 12KB of parsed floats. At 10,000 knowledge entries, that's ~120MB allocated on every search request. The function has no caching, no batching, and no early termination.

The hybrid search in `/api/agent/search` gracefully degrades when vector search fails (the `try/catch` around `searchByVector` swallows errors and falls back to BM25-only), so the immediate blast radius is limited to latency spikes and OOM under growth.

## Approval endpoint authorization gap

`POST /api/agent/approval` accepts `{ id, action, notes, resolvedBy }` from the request body. The `resolvedBy` field defaults to `"admin"` if not provided, but no verification occurs — the endpoint doesn't check headers, tokens, sessions, or any identity. An attacker who obtains or brute-forces a queue ID can approve stock mutations or follow-up creation without credentials.

The same pattern exists in `POST /api/agent/chat` where `approvalId` is passed to resume an approved flow. The lack of auth is consistent with the rest of the API surface (no routes in this project use auth), but the approval endpoint is the first route that mutates business state based on an untrusted caller's decision.

## Sub-agent infinite loop risk

Each sub-agent (CS, stock, follow-up) compiles a LangGraph `StateGraph` with a cycle: `llm -> tools -> llm`. The conditional edge after `llm` routes to `tools` if tool calls are present, and `tools` always routes back to `llm`. There is no `recursionLimit` passed to `.compile()` or `.invoke()`. LangGraph defaults to 25 iterations, which provides some protection, but the default is undocumented in this codebase and could change across library versions.

The single-agent graph similarly loops between `llm_decide` and `execute_tool` without an explicit cap, relying on the same library default. The `AGENT_MAX_ITERATIONS` env var is read in config but never used anywhere in the graph compilation or invocation.

```typescript
// config.ts reads this but nothing consumes it:
const maxIterations = parseInt(process.env.AGENT_MAX_ITERATIONS || "10", 10);
```

## Trace data model mismatch

`TracingHandler.saveTrace()` writes the full trace array as a JSON string into `AgentMetrics.errorMessage`:

```typescript
await prisma.agentMetrics.create({
  data: {
    requestId: this.requestId,
    sessionId: this.sessionId,
    stepName: "TRACE",
    errorMessage: JSON.stringify(this.trace),
  },
});
```

The `errorMessage` column is semantically for error strings. The metrics API filters out `stepName: "TRACE"` from aggregations, so traces don't corrupt error rate calculations, but the traces endpoint (`/api/agent/metrics/traces`) re-parses `errorMessage` as JSON, which will silently return `[]` if the JSON exceeds column limits or is malformed.

## Guard bypass in multi-agent sub-graphs

The multi-agent graph applies `guard_input` before routing and `guard_output` after sub-agent completion. However, the `guardConfig` loaded in `guard_input` is stored in the parent graph's state and never passed to the sub-agents. The sub-agents operate on `MessagesAnnotation` (messages only) and have no access to `guardConfig`. This means the sub-agents' tool calls and LLM responses are not individually guarded — only the final response leaving the parent graph passes through `guard_output`.

Tool side-effects (stock updates, follow-up creation) execute inside sub-agents without per-action permission checks from the guard. The `allowedActions` list in `guardConfig` is loaded but never consulted during tool execution in the multi-agent path. A guard config with `readOnlyMode: true` or a restricted `allowedActions` list would not prevent the sub-agents from mutating data.

## Memory consolidation deletes source records

The consolidation endpoint (`POST /api/agent/memory/consolidate`) promotes memories up the tier chain (working -> episodic -> semantic -> procedural) and then deletes the source records. If the LLM summarization produces a poor summary, there's no way to reconstruct the original working memories from the episodic summary. There's no soft-delete, no archive, and no rollback mechanism.

The deletion is done with `deleteMany` after the LLM calls complete, so a mid-process crash between `create` and `deleteMany` would duplicate rather than lose data. But a crash after `deleteMany` and before embedding generation would leave the promoted memory without an embedding, requiring a manual re-trigger of the embeddings endpoint.
