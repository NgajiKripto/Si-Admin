# Comprehensive security hardening for Si-Admin agent system

This PR closes 13 security vulnerabilities across authentication, authorization, input/output validation, and request integrity. The approach is defense-in-depth: missing auth guards are added to previously-open endpoints, the LangGraph agent is taught to distinguish authenticated from unauthenticated callers (blocking write tools for the latter), cumulative stock changes are persisted to the database to survive request boundaries, and a layered set of middleware checks (CSRF, SQLite blocking, rate limiting) harden the request pipeline. A knowledge content auditor gates knowledge base ingestion against prompt injection patterns.

**Watch for:** CSRF validation fails open when neither `Origin` nor `Referer` is present and `ALLOWED_ORIGIN` is not set (confirmed). Payload integrity check on approval is conditional on `payloadHash` being non-null, so pre-existing queue items bypass the check (confirmed). The knowledge batch endpoint has no `requireAuth` call despite other knowledge routes being guarded (confirmed).

## High-level view

Authentication gaps on memory endpoints, knowledge [id] routes, and the consolidation endpoint are closed by inserting `requireAuth` at the top of each handler.

The confused deputy fix splits the auth concern into two layers: chat routes remain open to unauthenticated users (so customers can use the chatbot), but the `executeToolNode` graph node checks `state.isAuthenticated` before allowing write tools. The read-only tool allowlist is defined once in `execute-tool.ts` with no enforcement that new tools default to restricted — though it fails safe (new tools are blocked), the list can silently drift from the actual tool inventory.

Cumulative stock tracking moves from in-memory graph state (which resets per request) to a persisted `cumulativeStockChanges` field on `AgentSession`. The approval threshold logic now reads from the database, making it resistant to the multi-request bypass.

The CSRF module uses Origin/Referer validation against the Host header, with an `ALLOWED_ORIGIN` override. When neither header is present and `ALLOWED_ORIGIN` is unset, the request is allowed through — CSRF protection is advisory rather than strict in the default configuration.

Rate limiting gains a multi-key check (`isRateLimitedMulti`) that records timestamps on both IP and session keys, so an attacker rotating IPs but reusing a session still gets throttled. However, each multi-key request contributes N timestamps toward the global 200-request cap, effectively halving it when 2 keys are used.

<details>
<summary>Issues (7)</summary>

1. **CSRF fails open without Origin/Referer** — When `ALLOWED_ORIGIN` is unset and neither `Origin` nor `Referer` is present, the CSRF check returns null (allow). Set `ALLOWED_ORIGIN` in production or change the default to reject when both headers are missing.
2. **Payload hash is optional** — Pre-existing approval queue items have `payloadHash: null`, so the integrity check is skipped. Consider rejecting approval of any item with a null hash, or backfill existing records.
3. **`.db` blocking over-matches** — The regex `/\.db/i` matches any path containing `.db` as a substring (e.g., `/api/something.dbreport` or paths with `.db` mid-segment). Use a more specific pattern like `/\.db($|[\?\/])/i` to anchor to actual file extensions.
4. **Read-only tool allowlist drift risk** — `READ_ONLY_TOOLS` in `execute-tool.ts` is a hardcoded array disconnected from tool definitions. New read-only tools will be silently blocked for unauthenticated users unless this list is updated. Consider co-locating the permission metadata with tool definitions.
5. **Knowledge batch endpoint has no auth** — The batch endpoint audits content for injection but doesn't call `requireAuth`. An unauthenticated caller can insert arbitrary (non-injection) content into the knowledge base.
6. **Multi-key rate limiter double-counts in global check** — `isRateLimitedMulti` records timestamps for all keys, then `isGloballyRateLimited()` sums all timestamps across all keys. Each 2-key request contributes 2 timestamps toward the 200-request global cap, reaching it at ~100 logical requests/minute.
7. **XSS escaping cascade** — Output validator XSS patterns run sequentially on `sanitizedOutput`. If an earlier pattern's escaping (replacing `<` with `&lt;`) partially overlaps with a later pattern's match region, the later pattern may not fire or may double-escape. Low probability but worth a test case with overlapping XSS patterns.

## CSRF validation bypass path

The fallback path in `validateCsrf` (when `ALLOWED_ORIGIN` is not set) returns `null` if both `Origin` and `Referer` are absent:

```typescript
// No Origin or Referer header - allow (some same-origin requests omit these)
return null;
```

A cross-origin attacker who can craft a request without `Origin` or `Referer` (e.g., via certain redirect chains or non-browser clients) bypasses CSRF protection entirely. The `ALLOWED_ORIGIN` path correctly rejects when both are missing, but the fallback path does not. In production, `ALLOWED_ORIGIN` must be set for the CSRF gate to be effective.

Confidence: **confirmed** — the code path is explicit at line 93 of `csrf.ts`.

## Knowledge batch endpoint missing auth guard

The batch endpoint (`/api/knowledge/batch`) imports `auditKnowledgeContent` but does not import or call `requireAuth`. The individual knowledge `[id]` routes do call `requireAuth`. An unauthenticated user can POST to the batch endpoint and insert knowledge entries as long as they pass the injection audit.

Confidence: **confirmed** — `requireAuth` does not appear in `src/app/api/knowledge/batch/route.ts`.

## `.db` path blocking false-positive surface

```typescript
const decodedPath = decodeURIComponent(request.nextUrl.pathname);
if (/\.db/i.test(decodedPath)) {
  return NextResponse.json({ error: "Not found" }, { status: 404 });
}
```

The regex matches any path containing the substring `.db` anywhere — including paths like `/reports/2024.dba-summary` or query-style paths where `.db` appears in a segment name. The intent is to prevent SQLite file download; the fix is to anchor the match to end-of-path or a path separator: `/\.db($|[\?\/])/i`.

Confidence: **confirmed** — regex lacks end-of-string or boundary anchor.

## Approval payload hash conditionality

```typescript
if (queueItem.payloadHash) {
  const computedHash = createHash("sha256").update(queueItem.actionPayload).digest("hex");
  if (computedHash !== queueItem.payloadHash) { ... }
}
```

The schema defines `payloadHash String?` (nullable). Queue items created before this migration, or by code paths that don't set the hash, skip the integrity check entirely. For a security control, this should reject when the hash is absent rather than silently allowing execution.

Confidence: **confirmed** — `payloadHash` is nullable in the schema, and the conditional is a no-op when null.

## Rate limiter multi-key global accounting

`isRateLimitedMulti` pushes a timestamp to every key in the array, so a request with 2 keys (IP + session) produces 2 timestamps in the global count. `isGloballyRateLimited` sums all timestamps across all keys. Under 2-key rate limiting, the effective global cap drops from 200 to ~100 logical requests/minute. This may trigger premature global throttling under legitimate load in a multi-user scenario.

Confidence: **likely** — based on reading the summing logic in `isGloballyRateLimited` and the double-push in `isRateLimitedMulti`.

## Test coverage gaps

Not tested: the confused deputy enforcement (unauthenticated user blocked from write tools), the cumulative stock DB persistence across requests, the `.db` path blocking in middleware, the SSE error sanitization in production mode, and the approval payload hash verification. These are the most critical security fixes in the PR and rely solely on code review for correctness.
