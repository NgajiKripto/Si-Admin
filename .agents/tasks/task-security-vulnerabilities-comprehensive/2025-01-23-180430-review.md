# Comprehensive security hardening for Si-Admin agent system (v2)

This PR closes 13 security vulnerabilities across authentication, authorization, input/output validation, and request integrity. Since v1 review, four of the seven flagged issues have been fixed: payload hash now rejects items without a hash, the `.db` regex is properly anchored, the knowledge batch endpoint has auth, and the multi-key rate limiter only records to the primary key.

**Watch for:** CSRF validation still fails open without Origin/Referer when `ALLOWED_ORIGIN` is unset (confirmed, documented trade-off). The read-only tool allowlist can drift from actual tool inventory (confirmed, fails safe). XSS escaping cascade remains theoretically possible but low-probability (confirmed).

**Verdict**: APPROVED

## High-level view

The approval endpoint now hard-rejects queue items lacking `payloadHash` — items created before this migration cannot be approved via the API, which is the safe default. The `.db` middleware regex is anchored with `($|[?/])` so it only fires on actual `.db` extensions. The knowledge batch endpoint now calls `requireAuth` before processing entries.

The rate limiter fix is the most structurally significant: `isRateLimitedMulti` checks all keys for violations but records a timestamp only on the primary key. This eliminates the double-counting that would have halved the global cap. The test "does not record timestamps when blocked" guards against regression.

Three issues from v1 remain as documented trade-offs: CSRF open-by-default without `ALLOWED_ORIGIN`, allowlist drift risk (fails safe), and XSS cascade (low probability). None are blocking.

<details>
<summary>Issues (3 remaining, non-blocking)</summary>

1. **CSRF fails open without Origin/Referer** — When `ALLOWED_ORIGIN` is unset and neither `Origin` nor `Referer` is present, the CSRF check returns null (allow). Set `ALLOWED_ORIGIN` in production. Non-blocking: the `ALLOWED_ORIGIN` path correctly rejects.
2. **Read-only tool allowlist drift risk** — `READ_ONLY_TOOLS` is a hardcoded array disconnected from tool definitions. Fails safe (new tools are blocked for unauthenticated users), but can silently restrict legitimate read-only tools. Non-blocking: operational, not security.
3. **XSS escaping cascade** — Sequential regex replacement could double-escape if patterns overlap. Low probability given the current pattern set. Non-blocking: worth a targeted test case in a follow-up.

## Fix verification: payload hash rejection

The approval route now rejects items with null `payloadHash` before attempting verification:

```typescript
if (!queueItem.payloadHash) {
  return NextResponse.json(
    { error: "Item ini tidak memiliki hash integritas. Silakan buat ulang aksi." },
    { status: 400 }
  );
}
```

Items without a hash cannot be executed — the admin must recreate the action to get a hashed payload. Confidence: **confirmed**.

## Fix verification: rate limiter single-key recording

`isRateLimitedMulti` now records a timestamp only for the primary key (first in array):

```typescript
// Record timestamp only for the primary key (first) to avoid global double-counting
const primaryKey = keys[0];
const primaryTimestamps = this.requests.get(primaryKey) || [];
primaryTimestamps.push(now);
this.requests.set(primaryKey, primaryTimestamps);
```

Secondary keys are checked for violations (if a session key is already exhausted, the request is blocked) but don't contribute timestamps to the global count. The effective global cap remains 200 logical requests/minute regardless of how many keys are used per request. Confidence: **confirmed**.

## Remaining: CSRF fallback path

The fallback path (no `ALLOWED_ORIGIN` env var) still returns `null` when both Origin and Referer are absent. In production, `ALLOWED_ORIGIN` must be set, which uses the strict path that rejects missing headers. The risk is limited to development/staging environments where `ALLOWED_ORIGIN` is unset. Confidence: **confirmed**.

## Test coverage gaps

Missing integration tests for the confused deputy path, cumulative stock DB persistence, `.db` middleware blocking, and approval hash verification. The rate limiter double-counting regression is now covered.
